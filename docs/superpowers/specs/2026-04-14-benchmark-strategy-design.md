# Benchmark Strategy Design

**Date:** 2026-04-14
**Status:** Draft
**Scope:** Local-only benchmark suite for ai-cortex — performance regression, quality validation, scale confidence

## Goals

1. **Regression detection** — catch perf degradations between commits
2. **Quality validation** — measure if suggest/blast_radius give correct/useful results
3. **Scale confidence** — prove it works on repos beyond the 165-file proof repo
4. **Local-first** — `pnpm bench` runs locally; graduate to CI later

## Architecture

Modular harness with pluggable suites. Separate modules for perf measurement, quality assertions, baseline comparison, and reporting.

### Directory Structure

```
benchmarks/
├── baselines.example.json    # committed structure template (no real numbers)
├── baselines.json            # local-only baselines per machine (gitignored)
├── config.ts                 # repo paths, SLOs, thresholds
├── runner.ts                 # CLI entry point (pnpm bench)
├── suites/
│   ├── perf-suite.ts         # timing measurements for all operations
│   └── quality-suite.ts      # golden sets + relative ranking assertions
├── fixtures/
│   └── synthetic/            # 50-file generated repo with known call chains
│       ├── generate.ts       # script to regenerate fixture
│       └── golden-sets.json  # expected results per task query
├── reporters/
│   ├── terminal.ts           # table output for local runs
│   └── json.ts               # JSON output for future CI
└── lib/
    ├── measure.ts            # timing harness (warmup, N runs, percentiles)
    ├── compare.ts            # baseline + SLO comparison logic
    └── types.ts              # shared types
```

### Package.json Scripts

```json
"bench": "tsx benchmarks/runner.ts",
"bench:perf": "tsx benchmarks/runner.ts --suite perf",
"bench:quality": "tsx benchmarks/runner.ts --suite quality",
"bench:update-baseline": "tsx benchmarks/runner.ts --update-baseline"
```

## Test Corpus

### Real Repos (perf + relative ranking)

| Repo        | Path                     | Size       | Required                     |
| ----------- | ------------------------ | ---------- | ---------------------------- |
| ai-cortex   | (derived from repo root) | ~86 files  | yes (self, always available) |
| ai-samantha | `~/Dev/ai-samantha`      | ~56 files  | no (discovered)              |
| ai-14all    | `~/Dev/ai-14all`         | ~142 files | no (discovered)              |
| ai-whisper  | `~/Dev/ai-whisper`       | ~218 files | no (discovered)              |

Only ai-cortex (the repo itself) is required by default. Its path is derived from the benchmark runner's repo root (e.g. `git rev-parse --show-toplevel` or relative to `import.meta.url`), not hardcoded. Other repos are discovered from `BENCH_REPOS` env var or `benchmarks/config.ts`. If absent, skipped with a warning — `pnpm bench` never fails due to missing external repos.

### Large Open-Source Repos (perf only, optional)

- 1-2 TS repos cloned into `benchmarks/fixtures/repos/` (gitignored)
- Fetched on first run or skipped if absent
- Config marks them `required: false` — `pnpm bench` never fails on a fresh clone

### Synthetic Repo (quality golden sets)

- 50 files committed to `benchmarks/fixtures/synthetic/`
- Structure: 3-4 modules (auth, api, db, utils), 10-15 files each
- Multiple call chains with branching, re-exports, barrel files
- Deterministic — no external deps
- Generated by `benchmarks/fixtures/synthetic/generate.ts`, output committed. Generator is run once to create the fixture; re-run only when fixture design changes

## Performance Suite

### Scenarios

Each scenario has explicit cache preconditions to ensure stable, reproducible measurements. The suite manages cache state (`~/.cache/ai-cortex/`) before each scenario.

| Scenario           | Operation            | Precondition                                               |
| ------------------ | -------------------- | ---------------------------------------------------------- |
| `index:cold`       | `indexRepo()`        | Delete cache before each run                               |
| `rehydrate:warm`   | `rehydrateRepo()`    | Ensure fresh cache exists (index once before measurement)  |
| `rehydrate:stale`  | `rehydrateRepo()`    | Index, then add an untracked file to force staleness check |
| `suggest:warm`     | `suggestRepo()`      | Ensure fresh cache exists, fixed task string               |
| `blastRadius:warm` | `queryBlastRadius()` | Ensure fresh cache with call graph, fixed target function  |

### Measurement Protocol

- 3 warmup runs (discarded)
- 20 measured runs (matches Phase 0 methodology)
- Record: p50, p95, min, max
- Each scenario measured per repo
- Cache state reset to precondition before each measured run where required (e.g. `index:cold` deletes cache each time; `rehydrate:warm` does not)

### Regression Check (relative)

- Compare p50 against `baselines.json`
- Fail threshold: >20% slower (configurable in `config.ts`)
- Warn threshold: >10% slower

### SLO Check (absolute)

```ts
slos: {
  "index:cold":        { small: 200, medium: 500,  large: 2000 },  // ms
  "rehydrate:warm":    { small: 20,  medium: 50,   large: 200  },
  "rehydrate:stale":   null,  // regression-only, no absolute SLO
  "suggest:warm":      { small: 50,  medium: 100,  large: 500  },
  "blastRadius:warm":  { small: 50,  medium: 100,  large: 500  },
}
```

SLO values are initial estimates — calibrate after first real run across all repos.

Scenarios with `null` SLO use regression checks only. `rehydrate:stale` triggers cache diffing and rebuild, making its cost highly variable by repo state — an absolute ceiling would be either too loose for warm or too strict for stale.

Repo size buckets:

- small: <100 indexable files
- medium: 100-500 indexable files
- large: >500 indexable files

### Baseline Management

- Baselines stored in `benchmarks/baselines.json` (gitignored, per-machine)
- `benchmarks/baselines.example.json` committed with structure template and zero values
- On first run with no baselines.json, suite warns and skips regression checks (SLO checks still apply)
- Updated explicitly via `pnpm bench --update-baseline`
- Baseline contains p50 per scenario per repo

## Quality Suite

### Golden Sets (synthetic repo)

Stored in `benchmarks/fixtures/synthetic/golden-sets.json`:

```json
{
	"suggest": [
		{
			"task": "fix the login token validation",
			"expected": ["auth/validate-token.ts", "auth/login.ts"],
			"limit": 5
		}
	],
	"blastRadius": [
		{
			"function": "hashPassword",
			"file": "auth/crypto.ts",
			"minConfidence": "full",
			"expectedHits": [
				{ "qualifiedName": "register", "file": "auth/register.ts", "hop": 1 },
				{
					"qualifiedName": "resetPassword",
					"file": "auth/reset-password.ts",
					"hop": 1
				},
				{
					"qualifiedName": "handleUserCreate",
					"file": "api/user-routes.ts",
					"hop": 2
				}
			]
		}
	]
}
```

**Metrics:**

- **Precision@k:** of top-k returned, how many are in expected set
- **Recall@k:** of expected set, how many appear in top-k
- Hard fail if precision@5 < 0.6 or recall@5 < 0.6 (tunable)
- Blast radius: assert each expected `(qualifiedName, file, hop)` tuple is present in result tiers. Fail if any expected hit missing. Also assert `confidence >= minConfidence` — catches regressions from newly unresolved edges.

### Relative Ranking (real repos)

Configured in `config.ts` per repo:

```ts
rankings: {
  "ai-cortex": [
    {
      task: "fix the suggest ranker scoring",
      higherThan: [
        ["src/lib/suggest-ranker.ts", "README.md"],
        ["src/lib/suggest.ts", "docs/shared/product_brief.md"],
      ]
    },
  ],
}
```

- Assertion: first file must rank higher than second
- ~3-5 ranking assertions per real repo
- Focus on obvious cases (implementation file > docs for code task)
- Only asserts relative order, not exact position — not brittle

```
── Quality ─────────────────────────────────────────────────
  suite         test                           status
  golden-set    fix login token validation     pass P@5=0.80 R@5=1.00
  golden-set    blast: hashPassword            pass 3/3 hits, confidence=full
  ranking       ai-cortex: fix suggest ranker  pass 3/3 pairs

── Summary ─────────────────────────────────────────────────
  perf:    8/10 pass, 1 warn, 1 skip (no baseline)
  quality: 8/8 pass
  result:  PASS (warnings only)
```

### Exit Codes

- `0` — all pass
- `1` — any fail (regression, SLO breach, or quality fail)
- Warns do not cause non-zero exit

### JSON Output

- `pnpm bench --json` writes to `benchmarks/results.json` (gitignored)
- Same data as terminal, structured for future CI consumption

## CLI Interface

```
pnpm bench                        # run all suites, compare against baselines
pnpm bench --suite perf           # perf only
pnpm bench --suite quality        # quality only
pnpm bench --update-baseline      # run perf + overwrite baselines.json
pnpm bench --json                 # output JSON alongside terminal
pnpm bench --repo ai-cortex       # run against single repo only
```

Runner parses `process.argv`, dispatches to suites, collects results, passes to reporter.
